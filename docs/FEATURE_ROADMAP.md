# 배드민국 완성 로드맵 — 필요한 것 전부

> 배드민국(badminguk.vercel.app) 기능 갭 조사 125건을 중복 통합·재조직한 완결 로드맵. 대상: 한국 배드민턴 MMR 플랫폼 (React 19 + Vite + Tailwind 4 + Supabase).

---

## 현황 진단 (읽고 시작할 것)

코드 감사 결과, **현재 배포본은 "보이기만 하는" 앱이다.** 다음 5가지가 깨져 있어 대회 하나를 실제로 완주할 수 없다:

1. `TEST_MODE=true` 하드코딩으로 인증이 전면 우회되고, 비로그인 상태에서 쓰기 동작이 전부 크래시하거나 RLS에 조용히 거부됨
2. 토너먼트가 1라운드만 생성되고 승자 진출 로직이 코드베이스에 없음 — 실제 완주 가능한 포맷은 리그전뿐
3. 대표 포맷 '조별리그+토너먼트' 엔진(`lib/tournament.js`)이 완성돼 있으나 어떤 페이지도 import하지 않음
4. 코트 관제(CourtView)·공개 스코어보드(LiveScore)가 존재하지 않는 컬럼을 조회해 영구 빈 화면
5. MMR 반영이 profiles RLS('본인만 수정')와 충돌해 실인증 환경에서 불가능하고, 반대로 match_scores는 아무 로그인 유저나 조작 가능

따라서 이 로드맵의 1단계는 "새 기능 추가"가 아니라 **"이미 있는 척하는 기능을 진짜로 만들기"**다.

## ① 대회 당일 운영 (최우선 카테고리)

### 1-1. 심판 태블릿 점수판 — BWF 규칙 엔진 내장 ★ [필수 / 대]
- **설명**: 코트별 심판이 태블릿/폰에서 1탭으로 랠리 점수를 입력하면 BWF 21점 랠리포인트 규칙이 자동 적용되는 패널. 20-20 듀스(2점차), 29-29 골든포인트, 게임·매치 종료 판정, 3게임째 11점 코트체인지, 서브권 전환까지 앱이 자동 처리. `games_per_match`(3판2선승)·`points_per_game`(21점) 설정을 실제 검증에 반영. 자동 심판 콜 문구("서비스 오버", "매치포인트") 화면 표시 + TTS 음성은 부가 옵션.
- **왜 필요한가**: 현재 점수 입력은 주최자 폰의 사후 세트스코어 일괄 입력뿐 — '실시간 스코어'가 실제론 결과 입력이다. BWF TSS 계열 오픈소스(bts)는 코트별 심판 패널 URL이 표준 구조이고, 국내 Score Syncer·위꾹라이브도 심판용 점수판을 핵심 상품으로 판매. 동호인 심판은 비전문가라 규칙 자동화가 실질 가치의 절반.
- **우선순위**: 필수 / **규모**: 대

### 1-2. 대진 진행 엔진 완성 (버그 수정) [필수 / 대]
- **설명**: (a) 토너먼트 2라운드 이후 경기 생성 + 승자 자동 진출 + 부전승 자동 처리 + roundLabel 역매핑 버그 수정, (b) 이미 완성돼 있으나 미연결인 `lib/tournament.js`(generatePools, calculatePoolStandings, determineAdvancements, generateKnockoutBracket)를 BracketGenerator에 연결해 대표 포맷 '조별리그+토너먼트'를 실제 작동시키고, 조별 순위표 화면 추가. 007의 tournament_pools 테이블 활용.
- **왜 필요한가**: 현재 `saveSchedule`의 `.filter(m => !m.bye)`가 2라운드+ 경기를 전부 버려 1라운드만 저장되고, CreateTournament가 수집한 조 편성 설정을 대진 생성기가 전부 무시한다. 완주 가능한 포맷이 리그전뿐인 상태.
- **우선순위**: 필수 / **규모**: 대

### 1-3. 코트 관제(CourtView)·공개 스코어보드(LiveScore) 재작성 (버그 수정) [필수 / 중]
- **설명**: 두 화면 모두 실제 스키마(category_id 경유, title, actual_start, 소문자 status)에 맞게 쿼리 전면 재작성. LiveScore는 고아 라우트 상태이므로 TournamentManage에 '공개 스코어보드 공유(QR/링크)' 버튼 추가.
- **왜 필요한가**: 가상의 스키마를 상대로 작성된 코드라 두 페이지가 항상 에러·빈 화면. /live/:id로 가는 링크가 앱 어디에도 없어 주최자가 관중에게 배포할 방법 자체가 없다.
- **우선순위**: 필수 / **규모**: 중

### 1-4. 포인트 단위 실시간 반영 파이프라인 [필수 / 중]
- **설명**: 심판 패널의 점수가 올라가는 즉시 공개 스코어보드·코트 관제·선수 앱에 Supabase Realtime으로 반영. 포인트별 이벤트 스트림 저장으로 랠리 그래프·리플레이 등 파생 기능의 토대 마련.
- **왜 필요한가**: Score Syncer·BWF 공식 라이브 스코어링 모두 point-by-point 스트림이 표준. Supabase Realtime으로 인프라 추가 없이 가능.
- **우선순위**: 필수 / **규모**: 중

### 1-5. 점수 이벤트 로그(append-only) + 언두 + 감사 추적 [높음 / 중]
- **설명**: 득점·정정·세트 종료를 모두 이벤트 행(match_events: 시각·입력자·종류·값)으로 적재하고 현재 스코어는 이벤트를 접어 계산. 잘못 누른 점수 1탭 되돌리기와 '누가 언제 무엇을 고쳤는지' 감사 로그가 자동으로 따라옴.
- **왜 필요한가**: 비전문 심판의 오입력은 반드시 발생하며(Umpire Score Keeper가 'undo with a single tap'을 대표 기능으로 명시), MMR이 걸린 플랫폼에서 점수 분쟁 시 근거가 됨. 오프라인 큐·충돌 병합(⑦)의 데이터 모델 기반이기도 함.
- **우선순위**: 높음 / **규모**: 중

### 1-6. 서브권·서비스 코트 표시 (코트 다이어그램) [높음 / 중]
- **설명**: 현재 서버, 좌/우 서비스 코트, 복식 점수 짝/홀 로테이션에 따른 선수 위치를 코트 모양 다이어그램으로 자동 추적·표시.
- **왜 필요한가**: 복식 서비스 순서는 동호인 심판이 가장 자주 헷갈리는 부분 — '누가 서브할 차례'를 알려주는 것이 심판 패널 실질 가치의 절반 (Umpire Score Keeper·bup의 핵심 기능).
- **우선순위**: 높음 / **규모**: 중

### 1-7. 인터벌·워밍업·부상 타이머 자동화 [높음 / 소]
- **설명**: 11점 도달 시 60초, 게임 사이 120초 인터벌과 워밍업·부상 치료 타이머를 심판 패널·관중 화면에 카운트다운으로 표시하고 종료 알림음으로 재개 독촉.
- **왜 필요한가**: BWF 경기규칙 명시 사항이며, 심판 앱들이 유료 기능으로 팔 만큼 수요 검증됨. 대회 전체 진행 지연을 줄이는 효과가 큼.
- **우선순위**: 높음 / **규모**: 소

### 1-8. 워크오버(W.O.)·기권(RET)·실격(DSQ) 정식 처리 [높음 / 중]
- **설명**: 경기 중 부상 기권, 불출전 워크오버, 실격을 정식 결과 상태로 기록하고 다음 라운드 진출·MMR 계산에 자동 반영. 현재 scheduled 상태에서만 노출되는 기권 버튼을 진행 중에도 허용, forfeit_reason 수집, 미사용 중인 `rescheduleAfterForfeit` 연결로 빈 코트 시간 자동 당김. MMR 정책(기권패 처리, 워크오버 레이팅 제외) 명시 필요.
- **왜 필요한가**: BWF RTTO가 공식 규정으로 두는 표준 플로우이며, 정책이 없으면 고의 기권으로 레이팅을 조작하는 구멍이 생김.
- **우선순위**: 높음 / **규모**: 중

### 1-9. 셀프 스코어링 모드 (심판 없는 코트, 상호 결과 확인) [높음 / 중]
- **설명**: 선수 본인들이 코트 옆 거치 태블릿/자기 폰으로 점수 입력. 경기 종료 시 양측이 각자 결과를 승인해야 확정, 불일치 시 운영석에 분쟁 알림.
- **왜 필요한가**: 동호인 대회는 전 코트 심판 배치가 불가능한 경우가 대부분(위꾹라이브가 소규모용 셀프 운영 시스템을 별도 상품화할 만큼 실수요 검증). MMR이 걸린 배드민국은 상호 확인 절차가 특히 중요.
- **우선순위**: 높음 / **규모**: 중

### 1-10. 대형 화면 디스플레이 모드 + 호출 전광판 + TTS 안내방송 [높음 / 중]
- **설명**: 코트별 전용 URL(/display/:tournamentId/:courtNo)을 TV·프로젝터 브라우저로 열면 초대형 고대비 점수판이 전체화면으로 표시. 게임 스코어·서브권·인터벌 카운트다운·다음 경기 예고 + 호출 큐 뷰("다음 입장: 홍길동·김철수 → 3번 코트")와 Web Speech API TTS 자동 음성 호출. 스폰서 배너 슬롯은 ⑧-4와 연결.
- **왜 필요한가**: bts의 표준 구조(심판 패널 /u2 ↔ 디스플레이 /d2)이며, 소음이 심한 체육관에서 마이크 육성 방송을 대체하는 마지막 안전망. 주최자의 '있어 보이는 대회' 욕구와 스폰서 노출 수익까지 연결.
- **우선순위**: 높음 / **규모**: 중

### 1-11. 스태프 권한 위임 + 코트 태블릿 인증 (PIN/QR) [중간 / 중]
- **설명**: 대회별 스태프 테이블(tournament_staff: referee/checkin 역할) + RLS 확장으로 주최자가 심판·진행요원에게 점수 입력·체크인 권한을 위임. 코트 태블릿은 QR/짧은 PIN으로 해당 코트 패널에만 접근, 누가 어느 경기를 심판했는지 기록.
- **왜 필요한가**: 현재 권한이 organizer_id 단일 계정에 종속돼 8코트 대회를 계정 하나로 돌려야 함. 점수 입력 권한 통제는 레이팅 신뢰도와 직결되며, 심판 점수판(1-1)의 전제 조건.
- **우선순위**: 중간 / **규모**: 중

### 1-12. 최종 순위·입상 발표·시상품 수령 관리 [높음 / 중]
- **설명**: 대회 completed 시 최종 순위 자동 산출, 입상자 발표 화면, 선수용 결과 페이지. 구현만 되고 미사용인 `prizeLabel`·prize_spots 연결. 시상품 등록과 QR/전자서명 수령 확인(종이 수령증 대체)까지.
- **왜 필요한가**: 주최자 여정의 '시상'과 선수 여정의 '결과 확인'이 모두 빈칸. 위꾹라이브가 '상품수령증 불필요'를 전면에 내세우고 배프는 상품관리 메뉴얼을 별도 제공할 만큼 실운영 수요가 큼.
- **우선순위**: 높음 / **규모**: 중

### 1-13. 경고카드(옐로·레드·블랙) 기록 [중간 / 소]

- **우선순위**: 필수 / **규모**: 중

### 3-3. 대회 탐색 포털·캘린더 (지역·급수·날짜 필터) [높음 / 중]

## ⑦ 기술 인프라

### 7-1. 오프라인 우선 점수 입력 (IndexedDB 큐 + Background Sync) [필수 / 대]
- **설명**: 점수 입력을 즉시 로컬 기록 + 낙관적 UI 반영 → Workbox/Serwist Background Sync 큐로 연결 복구 시 지수 백오프 자동 재전송.
- **왜 필요한가**: 체육관 와이파이는 악명 높게 불안정 — 네트워크 끊김 한 번에 점수판이 멈추면 현장 신뢰를 잃음. GameChanger 등 메이저 스코어링 앱의 핵심 기능.
- **우선순위**: 필수 / **규모**: 대

### 7-2. 점수 입력 충돌 해결 — 낙관적 동시성 + 입력 담당자 잠금 [필수 / 중]
- **설명**: 경기 레코드 version 컬럼 + `WHERE version=:읽은버전` UPDATE(불일치 시 거부·최신값 재표시), 경기별 '입력 담당자' 소프트 락 + 강제 인수 버튼. 점수를 append-only 이벤트 로그(①-5)로 저장하면 병합이 결정적.
- **왜 필요한가**: 관제와 심판 태블릿이 동시 수정하면 Last-Write-Wins로 점수가 조용히 왜곡됨. 오프라인 큐 도입 시 충돌 빈도가 급증하므로 7-1의 전제 조건.
- **우선순위**: 필수 / **규모**: 중

### 7-3. Supabase Realtime 신뢰성 보강 [필수 / 중]
- **설명**: `worker:true`(웹워커 heartbeat) + heartbeatCallback 연결 감시, 재연결 성공 시마다 REST 전체 재조회(refetch-on-reconnect), 실시간 핵심 채널은 postgres_changes 대신 broadcast 검토.
- **왜 필요한가**: Supabase Realtime의 silent disconnection과 재연결 중 이벤트 영구 유실은 공식 인정된 한계. 태블릿이 백그라운드로 가면 heartbeat가 밀려 끊김.
- **우선순위**: 필수 / **규모**: 중

### 7-4. 관중용 스코어보드 분리 — 폴링 + CDN 캐시 [높음 / 중]
- **설명**: WebSocket Realtime은 쓰기 주체(심판·관제)에만 할당, 관중용 /live/:id는 5~15초 폴링 + `s-maxage`/`stale-while-revalidate` CDN 흡수.
- **왜 필요한가**: Supabase Free는 동시 Realtime 연결 200개 한도 — 관중 수백 명이 열면 운영진 연결까지 끊김. 관중에게 15초 지연은 체감 안 됨.
- **우선순위**: 높음 / **규모**: 중

### 7-5. Screen Wake Lock — 태블릿 화면 꺼짐 방지 [높음 / 소]
- **설명**: 심판 점수판·코트 디스플레이에서 `navigator.wakeLock.request('screen')` + visibilitychange 재획득 + try-catch 상태 표시.
- **왜 필요한가**: 긴 랠리 중 절전 → 탭 비활성 → heartbeat 유실 → Realtime 끊김의 연쇄. 몇 줄로 해결되는 최고 가성비 항목.
- **우선순위**: 높음 / **규모**: 소

### 7-6. 동기화 상태 표시기 [높음 / 중]
- **설명**: 심판/관제 화면 상단에 연결 상태(실시간 연결됨/오프라인·전송 대기 N건/동기화 완료) 상시 표시, 대기 입력에 시계 아이콘. 관제에는 코트별 태블릿 마지막 통신 시각 표시(죽은 태블릿 즉시 발견).
- **왜 필요한가**: '내 입력이 서버에 갔는가'를 모르면 재입력·새로고침 반복 → 중복 데이터. silent disconnection은 상태 표시 없이는 인지 자체가 불가.
- **우선순위**: 높음 / **규모**: 중

### 7-7. PWA 설치 유도 (Android 버튼 + iOS 안내 오버레이) [높음 / 소]
- **설명**: beforeinstallprompt를 가로채 가치 경험 순간(참가신청 완료 직후)에 '앱으로 설치' 노출, iOS는 공유→'홈 화면에 추가' 안내 오버레이 별도 제작. manifest+SW는 7-1과 기반 공유.
- **왜 필요한가**: 재방문 경로 고정 + iOS 웹푸시의 전제 조건('홈 화면 추가' 후에만 권한 요청 가능). 한국은 iPhone 점유율이 높아 iOS 경로 누락 시 사용자 절반 손실.
- **우선순위**: 높음 / **규모**: 소

### 7-8. Vercel Hobby 탈출 + 한도 모니터링 [높음 / 소]
- **설명**: 수익화 시점에 Pro($20/월) 전환(Hobby는 약관상 상업 사용 금지 + 한도 도달 시 오버리지 없이 배포 정지), 그 전에도 사용량 알림 설정·정적 자산 최적화.
- **왜 필요한가**: 대회 당일 트래픽 스파이크로 한도를 치면 대회 도중 서비스가 통째로 내려가는 최악 시나리오.
- **우선순위**: 높음 / **규모**: 소

### 7-9. 태블릿 키오스크 모드 [중간 / 중]
- **설명**: 코트별 전용 토큰 URL(/court/:n/display, /court/:n/umpire) + Fullscreen API + 관리자 PIN 잠금, Android 앱 고정/iOS Guided Access 설정법을 주최자 준비 가이드로 제공.
- **왜 필요한가**: 관중·선수가 태블릿을 만져 화면 이탈하는 현장 사고 방지. bts의 검증된 URL 분리 구조와 동일.
- **우선순위**: 중간 / **규모**: 중

### 7-10. 리전 정렬 — Vercel 함수를 서울(icn1)로 [중간 / 소]
- **설명**: vercel.json regions를 icn1로 지정, x-vercel-id로 실행 리전 확인, DB 직결(5432) 대신 트랜잭션 풀러(6543) 사용 점검.
- **왜 필요한가**: 기본 리전(미국)이면 요청당 수백 ms 추가 + 스파이크 시 커넥션 고갈('too many connections'는 Next.js+Supabase 최다 장애 패턴).
- **우선순위**: 중간 / **규모**: 소

### 7-11. 대회 당일 부하 리허설 + 모니터링 런북 [중간 / 중]
- **설명**: k6로 관중 폴링+심판 동시 쓰기 리허설, Sentry(클라이언트 에러)·Supabase(Realtime 연결·DB 커넥션)·Vercel(대역폭) 대회 중 모니터링 런북, 재연결 지터(thundering herd 방지) 검증.
- **왜 필요한가**: 대회 당일은 평시 수십 배 트래픽이 3~4시간에 집중 — 현재는 터져야 아는 상태.
- **우선순위**: 중간 / **규모**: 중

### 7-12. OTP 발송 rate limit (버그 수정) [중간 / 소]
- **설명**: send-otp에 번호당 1분 쿨다운 + 일일 상한 + IP 제한.
- **왜 필요한가**: 현재 제한이 없어 악의적 반복 호출로 솔라피 SMS 과금 폭탄 가능, 재발송 무제한으로 시도 제한(5회) 브루트포스 리셋도 가능.
- **우선순위**: 중간 / **규모**: 소

## ⑧ 수익화

### 8-1. 결제 수수료 모델 (참가비의 3~5%) [높음 / 소]
- **설명**: '총 결제금액 − 플랫폼 수수료 = 정산금액'. 대진표·추첨·스코어보드는 무료로 열어 대회를 모으고 결제 흐름에서 수익. 영중소 우대 수수료(카드 1%대) 고려한 마진 설계.
- **왜 필요한가**: 이벤터스·온오프믹스가 검증한 국내 행사 플랫폼 표준 수익모델. 참가자 300명×5만원 대회면 수수료 4%로 60만원. 경쟁 배프는 사실상 무과금이라 선점 여지.
- **우선순위**: 높음 / **규모**: 소 (2-1 결제 위에 얹는 정책)

### 8-2. 주최자 정산 대시보드 + 토스 지급대행(서브몰) [높음 / 대]
- **설명**: 플랫폼이 참가비 수납 → 대회 종료 후 수수료 공제 지급. 핵심은 법적 구조: 직접 정산 관여 시 전자금융거래법상 PG 등록 대상이므로 토스페이먼츠 지급대행 API(주최자를 서브몰/셀러 등록, KYC 포함)로 정산을 외부 위탁해 등록 의무 회피. 대회별 수입 현황(신청·입금·환불·순수익) 대시보드.
- **왜 필요한가**: 금융위 유권해석상 정산 외부 위탁만이 안전한 구조이며, 티메프 사태 후 직접 자금 보관 모델은 규제 리스크 급증.
- **우선순위**: 높음 / **규모**: 대

### 8-3. 통신판매업 신고·현금영수증 컴플라이언스 온보딩 [높음 / 소]
- **설명**: 결제를 켜는 주최자에게 통신판매업 신고 확인 + 토스 '바로신청'(무료, 5분) 안내 플로우, 10만원 이상 현금성 결제(가상계좌 포함)의 현금영수증 자동 발급 설정.
- **왜 필요한가**: 결제 기능의 법적 전제조건. 복식 참가비 2인분이면 10만원 초과가 흔하고 미발급 가산세 20%.
- **우선순위**: 높음 / **규모**: 소

### 8-4. 스폰서 노출 패키지 [중간 / 소]
- **설명**: 공개 스코어보드·전광판 모드·추첨 애니메이션·대진표에 대회별 스폰서 로고/배너 슬롯. 주최자가 로고 업로드하면 체육관 대형화면·참가자 폰에 노출, 슬롯 개수·티어로 과금.
- **왜 필요한가**: 국내 동호인 대회는 용품사 협찬(빅터·테크니스트 콜라보)이 이미 표준. 개발 난도 대비 체감 가치가 가장 큰 수익 항목 — 주최자의 스폰서 유치 영업 도구가 됨.
- **우선순위**: 중간 / **규모**: 소

### 8-5. 선수용 프리미엄 구독 (MMR 상세 분석) [중간 / 중]
- **설명**: MMR 히스토리 심화, 파트너별/상대별 승률 분해, 급수 상승 예측, 시즌 리포트를 월 구독으로. 랭킹·기본 전적은 무료 유지(네트워크 효과 훼손 금지), '분석'만 유료.
- **왜 필요한가**: 이미 쌓이는 니치 데이터 자산의 원가 없는 프리미엄화 — 'AI commoditization moat' 원칙(해자는 니치 데이터)과 일치.
- **우선순위**: 중간 / **규모**: 중

### 8-6. 주최자 SaaS 티어 요금제 [중간 / 중]
- **설명**: 무료 티어(소규모 클럽 대회) + 유료 티어(참가자 무제한·커스텀 브랜딩·스폰서 슬롯·PASS 인증 건수·우선 지원).
- **왜 필요한가**: 무료 대회·계좌이체 고수 주최자에게서도 수익을 내는 보완 축. PASS 인증은 건당 실원가가 있어 유료 티어 편입 명분이 확실.
- **우선순위**: 중간 / **규모**: 중

### 8-7. 정산서·영수증 자동 발급 [중간 / 소]
- **설명**: 대회 종료 후 정산서(총 결제액·환불액·수수료·지급액) 자동 생성·발송, 참가자 결제 영수증.
- **왜 필요한가**: 협회·지자체 후원 대회는 회계 증빙 없이는 플랫폼 결제를 도입할 수 없음. 티메프 학습효과로 정산 투명성이 플랫폼 선택 기준이 됨.
- **우선순위**: 중간 / **규모**: 소

### 8-8. 상금 지급 관리 (원천징수 계산기) [낮음 / 소]
- **설명**: 부문별 상금 등록·지급 추적 + '기타소득 원천징수 22%, 공제 후 지급액 ○○원' 자동 계산·안내.
- **왜 필요한가**: 상금 대회가 늘고 있으나 세무를 아는 주최자는 드묾 — 세무 실수 방지가 곧 신뢰.
- **우선순위**: 낮음 / **규모**: 소

## 단계별 실행 순서

### 1단계 — "대회 하나를 처음부터 끝까지 돌릴 수 있게" (완주 MVP)

목표: 실제 대회 1개를 배드민국만으로 모집→추첨→진행→시상까지 완주. **여기까지 끝나야 어떤 마케팅도 의미가 있다.**

1. **TEST_MODE 제거 + 인증 정상화** (6-1) — 모든 것의 전제
2. **대진 진행 엔진 완성** (1-2) — 승자 진출 + 조별리그→토너먼트 연결
3. **심판 태블릿 점수판 1차** (1-1) — BWF 규칙 엔진 + 서브권 표시(1-6) + 언두(1-5 최소형). 태블릿 웹 패널로 시작
4. **실시간 파이프라인 + CourtView/LiveScore 재작성** (1-4, 1-3) — QR 공유 버튼 포함
5. **서버측 MMR 파이프라인 + RLS 봉합** (4-1, 6-9 일부)
6. **최종 순위·입상 발표** (1-12) — prizeLabel 연결
7. **모집 기본기**: 정원·자동 마감(2-4), 파트너 초대·수락(2-6), 셀프 취소(2-5 최소형), 대회 수정(2-9)
8. **워크오버·기권 처리** (1-8) — 현장에서 반드시 터지는 케이스
9. **당일 인프라 최소셋**: Wake Lock(7-5), Realtime 재연결 보강(7-3), 동기화 표시(7-6), 리전 정렬(7-10), OTP rate limit(7-12)
10. **법적 기반**: 약관·처리방침·동의 UI(6-2), 위탁 고지(6-3), 만14세 차단(6-4), 탈퇴·파기(6-5), 체크인 RLS 잠금(6-9)

### 2단계 — "운영을 자동화하고 돈이 돌게" (운영 완성 + 수익화)

목표: 주최자가 수작업 없이 200팀 대회를 굴리고, 플랫폼에 수익이 생긴다.

1. **결제 풀스택**: 토스 PG(2-1) → 무통장 자동확인(2-2) → 미납 자동취소·대기자 승격(2-3) → 환불 엔진(2-5 완성) → 컴플라이언스 온보딩(8-3) → 수수료 모델(8-1) → 정산 대시보드·지급대행(8-2) → 정산서(8-7)
2. **알림 풀스택**: 인프라(5-1) → 경기 호출(5-2) → 리마인더(5-4) → 공지 브로드캐스트(5-5) → 사전 예고(5-3) → 수신동의 레이어(5-8)
3. **당일 운영 심화**: 스태프 권한+태블릿 인증(1-11), 셀프 스코어링(1-9), 디스플레이 모드+전광판·TTS(1-10), 인터벌 타이머(1-7), 호출 ACK(5-6), 시상품 수령(1-12 완성), 경기 시간표(2-7)
4. **인프라 심화**: 오프라인 큐(7-1) + 충돌 해결(7-2) + 이벤트 로그 완성(1-5), 관중용 폴링 분리(7-4), PWA 설치 유도(7-7), Vercel Pro 전환(7-8), 키오스크 모드(7-9), 부하 리허설(7-11)
5. **신뢰 기반**: 백오피스(6-6), 이의신청+MMR 재계산(6-8), 샌드배깅 방지 기본(4-3: 디비전 캡·주최자 표출), 면책 동의서(6-10), 보험 표시(6-11), 실명 마스킹(6-12), CI/DI 대비(6-13)
6. **선수 기본 경험**: 내 경기 개인화 화면(3-1), 선수 전적 페이지(3-2), 대회 탐색 포털(3-3)

### 3단계 — "대회 도구에서 선수 플랫폼으로" (리텐션 + 차별화 + 확장)

목표: 대회가 없는 날에도 열어보는 앱. 경쟁사가 못 따라오는 MMR 데이터 해자 구축.

1. **급수 생태계 장악**: 급수 승급 자동 심사 엔진(4-2), 사기 신고·심사 워크플로(6-7), 콜드스타트 설문(4-15)
2. **랭킹 심화**: 페어 랭킹(4-7), 종목별 분리(4-8), 지역 랭킹(4-9), 신뢰도 점수(4-5), 감쇠(4-6), RD/Glicko-2(4-4), 연령대 랭킹(4-12), 리더보드 요건·강등 보호(4-13), 블로우아웃 제한(4-14), half-life(4-11), 시즌제(4-10)
3. **클럽·매칭 축**: 클럽 공간(3-4) → 정모 레크 모드(3-5) → 파트너/게스트 매칭(3-6) → 커뮤니티·채팅(3-7)
4. **포맷 확장**: 단체전 모드(2-8), 단식 완성(2-10), 클럽 분리 배치(2-11)
5. **수익 확장**: 스폰서 패키지(8-4), 프리미엄 구독(8-5), SaaS 티어(8-6), 상금 관리(8-8)
6. **부가 기능**: 영상 링크→VOD(3-8), 스코어시트 PDF(1-14), 경고카드(1-13), 레슨 매칭(3-9), 장비 기록(3-10), 셔틀콕 카운트(1-15), 친구톡 마케팅(5-9), LBS 대비(6-14)

### 원칙 요약

- **순서의 논리**: 1단계는 "있는 척하는 기능을 진짜로"(신뢰의 최소 조건) → 2단계는 "주최자의 수작업 제거 + 돈"(공급 측 락인) → 3단계는 "선수의 일상 침투"(수요 측 리텐션과 네트워크 효과).
- **심판 태블릿 점수판**은 1단계 3번으로 최우선 구현하되, 그 전제(인증·대진 엔진)와 후속(실시간 파이프라인·오프라인 내성)을 한 계보로 관리할 것.
- **MMR 데이터가 유일한 해자**다. 결제·알림·클럽은 경쟁사도 만들 수 있지만, 검증된 레이팅 + 급수 원장 + 샌드배깅 방지의 조합은 데이터가 쌓인 배드민국만 할 수 있다 — 3단계의 1~2번이 장기 승부처.
- 법규 항목(⑥)은 기능이 아니라 **기능을 켤 자격**이다. 결제는 컴플라이언스와, 알림은 수신동의와, PASS 실전환은 CI 고지와 반드시 한 몸으로 배포할 것.
